================================================================
 PYTHON PORT SCANNER — SCAN MODES EXPLAINED
================================================================

This scanner supports three scanning techniques, selected with
the -m / --mode flag. This document explains how each one works,
when to use it, and exactly how to run it.

LEGAL / ETHICAL NOTICE
-----------------------------------------------------------------
Only scan hosts you own or have explicit written permission to
test (e.g. a lab VM, a CTF box, or a target covered by an
authorized pentest engagement). Port scanning networks you don't
own or lack permission for can be illegal in many countries and
almost always violates acceptable-use policies. SYN scans in
particular are commonly flagged by firewalls, IDS, and IPS as
attack traffic — expect alerts if you run one against monitored
infrastructure without authorization.

The public host scanme.nmap.org is explicitly provided by the
Nmap project for practicing scans and is safe to use for testing.


================================================================
 1. CONNECT SCAN  (-m connect)  — default, no root required
================================================================

WHAT IT DOES
  Performs a full TCP three-way handshake against each port using
  a normal socket:
      SYN -----------------> (scanner)
      <----------- SYN/ACK   (target, if open)
      ACK -----------------> (target)
  The OS's socket.connect_ex() call handles all three steps for
  you. If the handshake completes (return code 0), the port is
  open. If the OS returns "connection refused," the port is
  closed. If nothing comes back before the timeout, the port is
  likely filtered by a firewall.

WHY USE IT
  - Doesn't need administrator/root privileges.
  - Most reliable and portable — works the same on every OS.
  - Lets you grab service banners (--banner), because the
    connection is fully established and you can read/write data.

TRADE-OFFS
  - Noisiest technique: every scanned port completes a full
    connection, which is trivially logged by the target and any
    firewall/IDS in front of it.
  - Slower than SYN scanning at very large scale, because full
    handshakes + OS-level connection teardown cost more than a
    single raw packet.

HOW TO RUN
    python3 scanner.py <target> -m connect -p 1-1024
    python3 scanner.py <target> -p 22,80,443 --banner -o results/scan.json

  (connect is the default, so "-m connect" can be omitted.)


================================================================
 2. SYN SCAN / "STEALTH SCAN"  (-m syn)  — requires root + scapy
================================================================

WHAT IT DOES
  Sends only the first packet of the TCP handshake (a SYN) using
  a raw socket, and inspects the reply — WITHOUT ever completing
  the connection:
      SYN -----------------> (scanner)
      <----------- SYN/ACK   => port is OPEN
                              (scanner then sends RST to tear the
                               half-open connection down cleanly,
                               instead of finishing the handshake)
      <----------- RST/ACK   => port is CLOSED
      <----------- (nothing) => port is FILTERED
                              (a firewall is likely silently
                               dropping the packet)

WHY IT'S CALLED "STEALTH"
  Because the three-way handshake is never completed, many
  simple/older logging setups that only record established TCP
  connections at the application layer never see it. This is a
  historical name, not a real guarantee — modern firewalls, IDS,
  and IPS (e.g. Snort, Suricata) detect and flag raw SYN floods
  just as easily as full connects, sometimes more so, because a
  burst of unanswered SYNs to many ports is itself a signature.

WHY USE IT
  - Faster for large port ranges since no full connection or
    teardown is needed per port.
  - Distinguishes open / closed / filtered more precisely than a
    connect scan sometimes can.

TRADE-OFFS
  - Requires raw-socket access, i.e. root/sudo on Linux/macOS or
    Administrator on Windows.
  - Requires the third-party `scapy` library (not in the standard
    library): pip install scapy
  - No banner grabbing — the connection is never fully opened, so
    there's no application data to read.
  - Can be treated as more "aggressive" by security tooling.

HOW TO RUN
    sudo python3 scanner.py <target> -m syn -p 1-1024
    sudo python3 scanner.py <target> -m syn -p 22,80,443 -t 50

  On Windows, run from an Administrator PowerShell/cmd prompt and
  make sure Npcap (https://npcap.com) is installed, since scapy
  needs it to send raw packets on Windows.


================================================================
 3. UDP SCAN  (-m udp)  — requires root for reliable results
================================================================

WHAT IT DOES
  UDP has no handshake, so "openness" has to be inferred:
    - Scanner sends an empty UDP datagram to the port.
    - If the service sends a UDP reply           => OPEN
    - If the OS/kernel gets back an ICMP "port
      unreachable" (type 3, code 3) error         => CLOSED
    - If nothing comes back before the timeout    => OPEN|FILTERED
      (ambiguous: the port could be open and simply not replying
       to an empty probe, or a firewall could be silently
       dropping the packet — there is no way to fully disambiguate
       this from the client side without service-specific probes)

WHY USE IT
  - Many important services run over UDP: DNS (53), NTP (123),
    SNMP (161), DHCP (67/68), and many game/streaming protocols.
    A TCP-only scan misses all of these entirely.

TRADE-OFFS
  - Inherently ambiguous (see OPEN|FILTERED above) — this is a
    property of UDP itself, not a limitation of this script. Even
    nmap reports the same "open|filtered" state for the same
    reason.
  - Slower in practice: many stacks rate-limit ICMP "unreachable"
    responses, so scanning many closed UDP ports can take longer
    than the same range over TCP.
  - Root is recommended so the OS reliably delivers ICMP
    unreachable errors back to the scanning socket; unprivileged
    UDP sockets on some OSes can miss or mis-report these.

HOW TO RUN
    sudo python3 scanner.py <target> -m udp -p 53,123,161
    sudo python3 scanner.py <target> -m udp -p 1-1024 -t 50 --timeout 2


================================================================
 QUICK REFERENCE
================================================================

  Mode      Flag         Root needed?   Speed        Notes
  --------  -----------  -------------  -----------  ------------------------
  connect   -m connect   No             Moderate     Default. Banner grabbing
                                                       works. Most reliable.
  syn       -m syn       Yes            Fast         Needs scapy. No banners.
                                                       Flagged by IDS/IPS.
  udp       -m udp       Recommended    Slow-ish     Results can be ambiguous
                                                       (open|filtered).

COMMON FLAGS (all modes)
  -p, --ports     Ports to scan: "80", "1-1024", or "22,80,443"
  -t, --threads   Concurrent workers (syn mode auto-caps at 20)
  --timeout       Per-port timeout in seconds (default 1.0)
  -o, --output    Save results as JSON
  -q, --quiet     Suppress the live progress counter
  --banner        (connect mode only) grab service banners

EXAMPLE WORKFLOW
  1. Quick sweep of common ports, full handshake, with banners:
       python3 scanner.py 10.0.0.5 -p 1-1024 --banner
  2. Fast stealth scan of the full port range:
       sudo python3 scanner.py 10.0.0.5 -m syn -p 1-65535 -t 100
  3. Check common UDP services the TCP scan can't see:
       sudo python3 scanner.py 10.0.0.5 -m udp -p 53,67,68,123,161
  4. Save everything to JSON for a report:
       python3 scanner.py 10.0.0.5 -p 1-1024 -o results/scan.json
